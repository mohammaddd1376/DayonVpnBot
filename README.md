# ربات تلگرامی فروش کانفیگ وایرگارد (NestJS)

ربات تلگرامی برای فروش کانفیگ وایرگارد. کاربران از طریق کیف پول داخلی خریداری می‌کنند
و شارژ کیف پول فقط به‌صورت دستی توسط ادمین انجام می‌شود (بدون درگاه پرداخت آنلاین).

## امکانات

**کاربر:**
- `/start` ثبت‌نام خودکار
- 🛒 خرید کانفیگ از بین پلن‌های فعال
- 👤 نمایش موجودی کیف پول
- 📂 نمایش لیست کانفیگ‌های خریداری‌شده + امکان حذف هر کدام

**ادمین** (فقط شناسه‌های تلگرام موجود در `ADMIN_TELEGRAM_IDS`):
- `/charge <شناسه_عددی> <مقدار_تومان> [توضیح]` — شارژ دستی کیف پول کاربر
- `/addplan نام|قیمت|روزاعتبار|توضیح` — افزودن پلن جدید
- `/plans` — لیست همه پلن‌ها
- `/deactivateplan <شناسه_پلن>` — غیرفعال کردن یک پلن
- `/finduser <شناسه_عددی>` — مشاهده موجودی و تعداد کانفیگ‌های یک کاربر

## پیش‌نیاز

- Node.js نسخه ۱۸ یا بالاتر
- PostgreSQL (یا از `docker-compose.yml` همراه پروژه استفاده کنید)
- توکن ربات از [@BotFather](https://t.me/BotFather)

## نصب و اجرا

```bash
npm install
cp .env.example .env
```

فایل `.env` را ویرایش کنید:
- `BOT_TOKEN` — توکن دریافتی از BotFather
- `ADMIN_TELEGRAM_IDS` — شناسه عددی تلگرام خودتان (با /start در ربات @userinfobot قابل دریافت است)
- اطلاعات دیتابیس را مطابق محیط خودتان تنظیم کنید

اگر postgres ندارید:
```bash
docker compose up -d
```

اجرای ربات در حالت توسعه:
```bash
npm run start:dev
```

اجرای در پروداکشن:
```bash
npm run build
npm run start:prod
```

## مهم: هماهنگ‌سازی با API واقعی سرور وایرگارد

این پروژه با دو endpoint زیر صحبت می‌کند:
```
GET http://51.194.215.156:9299/create?publicKey=NAME
GET http://51.194.215.156:9299/remove?publicKey=NAME
```

چون فرمت دقیق پاسخ این دو endpoint مشخص نبود، در فایل
`src/vpn-configs/vpn-api.client.ts` فرض شده پاسخ `/create` همان محتوای کانفیگ
(متن خام) است که در قالب فایل `.conf` برای کاربر ارسال می‌شود. اگر فرمت واقعی
پاسخ سرورتان متفاوت است (مثلاً JSON با فیلدهای دیگر)، فقط همین یک فایل را اصلاح
کنید؛ بقیه پروژه (کیف پول، پلن‌ها، ربات) به این جزئیات وابسته نیست.

## نکات مهم برای استفاده در محیط واقعی (Production)

1. `DB_SYNCHRONIZE=true` فقط برای توسعه مناسب است. در پروداکشن باید آن را `false`
   کنید و از TypeORM migrations استفاده کنید تا تغییرات schema کنترل‌شده باشد.
2. وضعیت گفتگوهای چندمرحله‌ای (در این نسخه استفاده نشده، چون دستورات ادمین به‌صورت
   تک‌خطی طراحی شده‌اند) به‌صورت پیش‌فرض ذخیره نمی‌شود؛ اگر بعداً سناریوهای چندمرحله‌ای
   اضافه کردید، برای مقیاس بالا session را روی Redis نگه دارید نه حافظه.
3. توصیه می‌شود ربات را با `pm2` یا داخل یک کانتینر Docker با `restart: always`
   اجرا کنید تا در صورت کرش مجدد بالا بیاید.
4. در صورتی که چند نسخه از ربات را هم‌زمان اجرا کنید (مثلاً برای HA)، باید از
   webhook به‌جای polling استفاده کنید تا تلگرام خطای ۴۰۹ ندهد.

## ساختار پروژه

```
src/
  users/        مدیریت کاربران
  wallet/       کیف پول و تراکنش‌ها (شارژ دستی، کسر هنگام خرید)
  plans/        پلن‌های قابل خرید
  vpn-configs/  ارتباط با API سرور وایرگارد + ذخیره کانفیگ‌های خریداری‌شده
  bot/          هندلرهای ربات تلگرام (Telegraf)
  common/       گاردها (مثلاً تشخیص ادمین)
```
